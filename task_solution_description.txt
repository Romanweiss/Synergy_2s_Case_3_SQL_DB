Подробное описание процесса решения задачи (разработка с нуля)
1. Анализ постановки задачи
- Требовалось спроектировать БД предметной области «Туризм».
- По условию нужно создать 4–5 таблиц, из них 3–4 справочника и 1 таблицу переменной информации.
- Для всех таблиц должны быть первичные ключи, а таблица переменной информации должна ссылаться на справочники через внешние ключи.

2. Выбор СУБД и формата результата
- Для реализации выбрана PostgreSQL.
- Результат оформлен в виде SQL-скрипта `tourism.sql` (DDL + ограничения + индексы + тестовые данные).
- Такой формат полностью соответствует требованию предоставить SQL-скрипт с определениями структуры базы данных.

3. Проектирование сущностей
- Определены 4 таблицы-справочника:
  - `clients` (клиенты);
  - `tours` (туры);
  - `services` (услуги);
  - `managers` (менеджеры).
- Определена 1 таблица переменной информации:
  - `orders` (заказы туров).

4. Проектирование ключей и связей
- Во всех таблицах созданы первичные ключи (`PRIMARY KEY`).
- В таблице `orders` созданы внешние ключи (`FOREIGN KEY`) на все справочники:
  - `client_id` -> `clients.client_id`;
  - `tour_id` -> `tours.tour_id`;
  - `service_id` -> `services.service_id`;
  - `manager_id` -> `managers.manager_id`.
- Добавлены ограничения целостности (`NOT NULL`, `CHECK`, `UNIQUE`) для контроля корректности данных.

5. Модель данных и ограничения
- Для идентификаторов использованы автоинкрементные поля (`GENERATED ALWAYS AS IDENTITY`).
- Для денежных значений использован тип `NUMERIC(12,2)`.
- Для статуса заказа добавлен контроль допустимых значений через `CHECK`.
- Для предотвращения дублей в заказах добавлено бизнес-ограничение `UNIQUE`.

6. Оптимизация и проверка
- Для таблицы `orders` добавлены индексы по внешним ключам и часто используемым полям (`status`, `start_date`).
- Добавлены тестовые данные (`INSERT`) для проверки связей и корректности ограничений.

7. Документирование
- Обновлён `README.md` с описанием структуры БД и подтверждением соответствия условию.
- Добавлена схема `tourism_schema.png` для визуальной проверки модели.
